# 协作者手册

## 可以贡献什么

- 往年试卷 → `exam/<课程名>/`，记得写答案（可以直接让 AI 生成）
- 课后作业 → `homework/<课程名>/`
- 课程讲义与资料 → `lecture/<课程名>/`（目前为空目录，欢迎大家补充）
- 命名示例：`高等数学A（II）-25春-期末.tex`、`第3次作业.tex`

## 排版规则

- 每个 .tex 文件开头写 `\heading{课程名-说明}`，用 `question` 环境包裹题目，`solution` 环境包裹解答
- 配图用 `\pyfig[宽度]{路径}{说明}`，图放在对应课程的 `figures/` 文件夹
- 不要写 `\noteinfo`、`\bigskip`、`\ifShowSolutions`。不用管 `\begin{document}`，每个 .tex 只是一段内容

## 工作流程

**请在独立分支上开发，不要直接往 master 提交。**

### 步骤

#### 1. 创建自己的分支

```bash
# 务必先拉取最新 master，确保你的起点是最新版
git checkout master
git pull --rebase

# 创建新分支（以你的改动命名）
git checkout -b feat/添加热学试卷
```

分支命名建议：
- `feat/添加xxx` — 新增内容
- `fix/xxx` — 修复问题
- `docs/xxx` — 文档改动

#### 2. 开发并提交

```bash
git add -A
git commit -m "feat: 添加了……"
git push origin feat/添加热学试卷
```

#### 3. 创建 Pull Request

```bash
gh pr create --base master --title "你的改动" --body "简单描述改了什么"
```

或者在 GitHub 上打开 https://github.com/claudemt/CourseMaterial 后会出现 Pull Request 提示。

#### 4. 等待合并

管理员看到 PR 后，没问题就会合并。如果有需要修改的地方会在 PR 里留言。

#### 5. 如果 master 有新进展，需要同步

```bash
git checkout feat/添加热学试卷
git pull --rebase origin master   # 将 master 的最新改动同步到你的分支
git push origin feat/添加热学试卷 --force-with-lease
```

#### 6. 合并后清理

PR 合入后，删除本地分支：

```bash
git checkout master
git pull --rebase
git branch -d feat/添加热学试卷
```

## ⚠️ 防冲突指南（必读）

> 以下内容基于真实教训总结，**每次提交前请过一遍**。

### 问题 1：从旧版本拉分支

**❌ 错误做法：**
```bash
# 没有先拉最新 master，直接从本地旧版本拉了分支
git checkout -b feat/xxx  # ← 你的 master 还是两周前的版本！
```

**✅ 正确做法：**
```bash
# 先让 master 保持最新，再拉分支
git checkout master
git pull --rebase          # ← 这步不能省
git checkout -b feat/xxx
```

> 💡 **为什么重要：** 仓库一直在更新，你拉的起点越旧，后面冲突越多。
> 比如你拉分支时还没 `homework/理论力学/` 这个目录，你在分支上创建了同名文件，
> 合并时就会冲突。

---

### 问题 2：开发期间从不同步 master

**❌ 错误做法：**
在分支上写了一周代码，从不拉取 master 的更新，最后发现完全跟 master 脱节了。

**✅ 正确做法：**
```bash
# 建议每隔几天同步一次，避免积攒太多冲突
git pull --rebase origin master
```

> 💡 **为什么重要：** 冲突越早解决越小，越晚解决越大。
> 每天花 10 秒同步一次，比最后花一小时解决冲突划算得多。

---

### 问题 3：误删了别人新加的文件

这是最常见的坑！来看真实案例：

```
你拉分支时 master 的状态：
  homework/
    ├── 数学物理方法（上）/     ✅
    └── 实验物理中的统计方法/   ✅

你本地没有（当时 master 还没有）：
  homework/理论力学/          ← 你根本看不到，但别人已在 master 加了

你在分支上做了：
  rm -rf homework/理论力学/   ← ❌ 你看着本地没有就删了，但 master 有啊！
```

**✅ 原则：**

| 原则 | 说明 |
|------|------|
| **只增不删** | 除非你 100% 确定某个文件不再需要，否则**只加新文件，不动旧文件** |
| **不要用 `rm` 命令** | 只用 `git add` 添加你的新文件，不要手动跑 `rm` 或 `git rm` |
| **不碰无关目录** | 你只负责某个课程，就别去动其他课程的文件夹 |
| **提交前检查改动** | 用下面命令看自己到底改了啥（见"提交前自检"） |

> 拿不准某文件该不该删？在 PR 里问一下，不要自己直接删。

---

### 问题 4：提交前不检查

**提交前必须检查你的改动范围：**

```bash
# 看看你改了什么（和 master 对比）
git fetch origin
git diff origin/master --name-status
```

这个命令会列出你所有**相对于远程最新 master** 的改动。逐条检查：
- ✅ `A  homework/热学/第1次作业.tex` — 对，这是我要新增的
- ❌ `D  docs/协作者手册.md` — 等等，我怎么会删了这个？！
- ❌ `M  exam/某课程/期末.tex` — 我根本没动过这个文件啊？！

如果看到 `D`（删除）或 `M`（修改），但并不是你刻意修改的文件——说明你的分支起点太旧了，需要先同步 master。

### 提交前自检清单

在 `git push` 之前，花 30 秒跑一遍：

```bash
# 1. 拉取最新 master
git fetch origin

# 2. 看看你的分支跟 master 差多少
git log --oneline origin/master..HEAD

# 3. 看看你改了哪些文件（重点关注 D=删除、M=修改）
git diff origin/master --name-status
```

**检查点：**
- [ ] 我的分支是从最新 master 拉的吗？
- [ ] 我的改动只涉及我想改的文件吗？
- [ ] 有没有我不小心删除的文件？（`D` 开头的行）
- [ ] 有没有我不小心修改的文件？（`M` 开头但没印象的）
- [ ] 如果有 `README.md` 的修改，是故意改的吗？（README 很容易被不小心改到）

### 如果发现有问题怎么办？

```bash
# 你的分支和 master 已经不同步了，需要 rebase
git pull --rebase origin master

# 如果 rebase 过程中有冲突，git 会提示你手动解决
# 解决完后：
git add -A
git rebase --continue
git push origin feat/xxx --force-with-lease
```

### 冲突产生时怎么解决？

如果 `git pull --rebase origin master` 提示冲突：

```
CONFLICT (add/add): Merge conflict in exam/xxx.tex
```

**解决步骤：**

```bash
# 1. 打开冲突文件，搜索 <<<<<<< 标记
#    上半部分是 master 的版本，下半部分是你的版本
#    修改成你想要的样子，删掉标记行

# 2. 标记为已解决
git add 冲突文件.tex

# 3. 继续 rebase
git rebase --continue

# 4. 推送
git push origin feat/xxx --force-with-lease
```

如果觉得搞不定，直接暂停 rebase，来问：

```bash
git rebase --abort   # 回到 rebase 之前的状态
```

---

### 一句话总结

> **每次做 git 操作前，先 `git pull --rebase origin master`；操作后，先 `git diff origin/master --name-status` 再 push。**
